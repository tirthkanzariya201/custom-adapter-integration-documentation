Phase 1 — Snippet rollout runbook

For Tirth · Scope: 5 pure-prose blocks, no variables · Goal: kill the highest-duplication drift with near-zero risk
VERIFIED (JamDesk docs) — Snippets live in /snippets/*.mdx and are inserted with <Snippet file="name.mdx" />. Pages are MDX; nav is in docs.json. Deploy is jamdesk validate → jamdesk deploy (your confirmed manual workflow — git push alone doesn't publish for your project).

Golden rule: pilot one, then scale

Prove the mechanism on one snippet, one page and deploy it before touching 30 pages. If the pilot renders correctly live, the pattern is safe to repeat mechanically.

The repeatable procedure (per snippet)

  1. Create the snippet file. Add snippets/<name>.mdx containing only the canonical block (no title/description frontmatter — snippets are fragments, not pages).
  2. Replace one inline copy. On the pilot page, delete the inline block and insert <Snippet file="<name>.mdx" /> in its place.
  3. Validate: run jamdesk validate — catches broken references and MDX errors before deploy.
  4. Deploy & eyeball: jamdesk deploy, then open the live pilot page and confirm the block renders identically to before.
  5. Scale: once confirmed, repeat step 2 (delete inline → insert the same <Snippet> tag) on every other page carrying that block. Validate, deploy.

Verify before you start (don't assume)

  • Is <Snippet> globally available or does it need an import line at the top of each MDX file? Confirm on the pilot page — if the pilot renders without an import, it's global.
  • Exact local-preview command. I know jamdesk validate and jamdesk deploy are real; I'm not certain of the local live-preview command — check jamdesk --help so Tirth can preview before deploying rather than eyeballing production.
  • Gradle block (#4): confirm it's actually duplicated on Android pages vs already linked to the Gradle Setup page. If it's already a link, skip it — it's done.

Most cost-effective way to run the bulk edits

Don't do the ~30-page find-and-replace by hand, and don't run it through the chat interface (expensive, slow). Phase 1 done = definition

All 5 blocks exist once in /snippets; every page references them; jamdesk validate passes; a spot-check of one page per framework renders correctly. Then — the gate: agree that any future edit to these blocks happens in the snippet, never inline, or the drift comes back.

Not in Phase 1: values (class strings, keys, versions). Those are Phase 2 data-reference, and I'm not certain JamDesk snippets accept props/variables the way Mintlify-style platforms do — verify JamDesk's variables mechanism before scoping Phase 2. Phase 1 needs none of that.
