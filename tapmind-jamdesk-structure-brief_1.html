How the JAMdesk integration doc is built

The complete structure — platform model, organizing system, modules, and the path from edit to live — for planning and tracking the rebuild of docs.tapmind.io.

For: John Britto (PM) Covers: CA-GMA · CA Next-Gen · Orchestration Platform: JAMdesk (Git-backed MDX)

1 · The platform

How JAMdesk works (the model to hold in your head)

JAMdesk is not a CMS you log into and type pages — it is a code-backed docs site. The whole site is a folder of text files in a GitHub repo; pushing to the repo rebuilds and publishes the site. Five facts cover it:

1Pages are files

Each page is an .mdx file (Markdown + reusable components) in the repo. No database.

2One nav config

docs.json defines the entire sidebar — order, grouping, tabs. Change structure here, not in pages.

3Reusable snippets

Shared blocks live in /snippets and are imported into pages. Write once, appears everywhere.

4Auto AI feed

JAMdesk auto-generates llms.txt so AI tools read the live docs — no extra work.

5Git = publish

Push to GitHub → JAMdesk builds → live on docs.tapmind.io. Preview locally before pushing.

Launch gate: the site is currently behind a login wall (returns “401”). Publishers must reach it without our credentials. Confirm the access model is set to public before launch — an integration doc readers can’t open is a non-starter.

JAMdesk is a Mintlify-family toolchain; exact CLI commands and snippet capabilities should be confirmed against JAMdesk’s own docs before relying on them.

2 · The organizing model

Funnel → Path → Loop

A reader doesn’t think in pages; they think “I am this kind of integrator, on this framework, using this mediation.” The doc is built to route them down one line and run the same shape every time. This single idea drives every structural decision below.

FUNNEL

Narrow the reader

Front-door pages move them from “what is this” → which SDK → which framework + mediation, before any code.

PATH

One page per cell

Each framework × mediation × SDK combination is one page. The reader follows exactly one.

LOOP

Same 6 steps, always

Every page runs the identical 6-step sequence — so once you’ve done one, you can do any.

Today’s doc is the opposite: 33 near-duplicate pages, each assembled differently, with no front door. That inconsistency is why wrong values (a typo’d class, a transposed key) spread across the fleet unnoticed.

4 · The module system

Pages, snippets, and the anti-drift principle

The core engineering idea: every fact lives in exactly one place. A page is thin; anything repeated across pages is a snippet that pages import. The class string and network key live in one snippet each — fix it once, all pages update. This is the structural fix for how a single typo reached 30 pages.

repo/ docs.json # the sidebar (Section 3) index.mdx concepts.mdx choose-your-sdk.mdx prerequisites.mdx guides/{framework}/{product}/{mediation}.mdx # the 33 thin pages — each is the 6-step Loop reference/ class-network-key-registry.mdx # human-readable source of truth compatibility-matrix.mdx troubleshooting.mdx privacy-consent.mdx snippets/ config/ admob-dashboard · max-dashboard · levelplay-dashboard · gam-dashboard ↑ hold the dashboard steps + the canonical class string / network key shared/ prerequisites-* · android-gradle-repos · app-id-* · verify-ad · troubleshoot ↑ blocks repeated across many pages (install repo block, verify step, etc.)

Why this matters to you as PM: it converts a recurring quality problem into a one-line fix. When the dev confirms the correct class string, it’s changed in a single snippet — not hunted across 30 pages. It also means the bulk of the “rebuild” is wiring, not writing.

5 · Page anatomy

The 6-step Loop — every mediation page, same order

Each page runs these six steps. Steps are tagged by who performs them, so the developer and the ad-ops operator can each skip to their part — something none of our competitors’ docs do cleanly.

Only Step 2 is unique per page (the install line). Step 4 is the imported dashboard snippet that carries the class string. Steps 5–6 close the loop and catch failures — both missing from today’s doc. Writing a new page = copy the template, fill the install line, swap one import.
