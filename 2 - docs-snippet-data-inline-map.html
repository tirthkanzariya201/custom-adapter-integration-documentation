Keep inline — genuinely unique

Very little survives here once you decompose. Mainly: the page's assembly/order, and any true one-off note for that exact framework×mediation cell. If a page has more than a few lines of unique prose, that's a signal it's carrying something that should have been a snippet.

Lean rollout — highest ROI, lowest risk first (don't boil the ocean)

  1. Confirm current state first (dev-team, 2 min): diff the MDX source of two AdMob pages. Snippets already in use, or raw duplication? Don't refactor blind.
  2. Phase 1 — the safe, high-duplication wins: placementName callout, SKAdNetwork list, Gradle repos, test-mode note, troubleshooting link. Pure prose, identical everywhere, zero logic. Immediate drift kill, near-zero risk.
  3. Phase 2 — data-reference the values: point class strings / keys / versions at the Registry + Matrix so the values you've already seen drift (Flutter LevelPlay 1.0.0, the network key) can't diverge again.
  4. Phase 3 — the structural snippets: per-framework install blocks and per-mediation dashboard blocks. Higher payoff, but touches page assembly, so do it after the easy wins prove the mechanism.
What to think about that you're not asking: the end state — a page as composition of snippets + data — is one step short of generating pages from the release registry. You already have the registry pieces in flight (Jira → GitHub → JamDesk). Building from scratch, you can make docs a build artifact of the release: cut a version, pages regenerate, drift becomes structurally impossible. Incumbents with hand-written legacy doc estates can't cheaply get there. But don't build the generator now — Phases 1–2 are lean and ship this week; just structure the snippets/data so a generator can consume them later without a rewrite. Extensible, not over-engineered.

Verify: I'm not certain of JamDesk's exact snippet syntax (MDX import vs a <Snippet> component vs a /snippets convention) — confirm in JamDesk's docs before scoping. This whole map assumes JamDesk supports transclusion + a values/frontmatter source it can read; if it can't do data-reference natively, the "values" rows may need a small build step.
